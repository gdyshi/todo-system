# Todo System 项目维护工作流

本文档定义了项目的标准 GitHub 工作流程。

---

## 标准工作流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                        开发阶段 (本地)                               │
├─────────────────────────────────────────────────────────────────────┤
│  1. 创建功能分支 (feature/* 或 fix/*)                               │
│  2. 本地编码修改                                                    │
│  3. 本地测试 (pytest + playwright)                                 │
│  4. 推送到 GitHub                                                    │
└─────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────┐
│                        PR 阶段 (自动)                                │
├─────────────────────────────────────────────────────────────────────┤
│  ✓ 代码质量检查 (Black, Flake8, MyPy)                               │
│  ✓ 单元测试 + 集成测试                                               │
│  ✓ 安全扫描 (Trivy)                                                 │
│  ✓ Preview 部署 (前端)                                              │
│  ✓ 轻量 E2E 测试 (关键路径)                                          │
│  ✓ 检查结果 PR 评论                                                 │
└─────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────┐
│                        审查阶段                                      │
├─────────────────────────────────────────────────────────────────────┤
│  Reviewer 检查：                                                     │
│  - 代码逻辑                                                         │
│  - Preview 效果（点击链接直接看）                                     │
│  - 测试覆盖率                                                       │
│  - 至少 1 人批准                                                    │
└─────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────┐
│                      合并阶段 (master)                               │
├─────────────────────────────────────────────────────────────────────┤
│  ✓ 所有检查通过                                                      │
│  ✓ 至少 1 个 Reviewer 批准                                           │
│  ✓ Squash and Merge 到 master                                        │
│  ✓ Staging 环境自动部署                                              │
│  ✓ 完整 E2E 测试自动运行                                             │
│  ✓ 性能基准测试                                                      │
└─────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────┐
│                      生产部署 (手动)                                 │
├─────────────────────────────────────────────────────────────────────┤
│  1. 确认 Staging E2E 测试通过                                        │
│  2. 触发 deploy-production workflow                                  │
│  3. 选择部署目标 (staging/production)                                │
│  4. 确认执行                                                        │
│  5. 部署 + 健康检查                                                  │
│  6. 完成通知                                                        │
│  7. 如有问题 → 执行 rollback                                         │
└─────────────────────────────────────────────────────────────────────┘
```

## 分支策略

### 分支命名规范

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 功能 | `feature/<name>` | `feature/add-user-auth` |
| 修复 | `fix/<name>` | `fix/login-crash` |
| 测试 | `test/<name>` | `test/e2e-workflow` |
| 文档 | `docs/<name>` | `docs/update-readme` |

### 分支保护规则

**master 分支**已设置以下保护规则：
- ✅ 需要 Pull Request 才能合并
- ✅ 需要 1 人审查批准
- ✅ 新提交时自动清除旧的审查
- ❌ 不强制管理员遵守规则

## 本地开发流程

### 1. 创建功能分支

```bash
# 确保本地是最新的
git checkout master
git pull origin master

# 创建功能分支
git checkout -b feature/my-feature
```

### 2. 本地编码

```bash
# 编码...
# 定期提交
git add .
git commit -m "feat: 实现某功能"
```

### 3. 本地测试

```bash
# 后端单元测试
cd backend
pytest tests/ -v

# E2E 测试 (需要启动服务)
cd ..
npx playwright test

# 代码格式检查
black --check .
flake8 .
mypy .
```

### 4. 推送并创建 PR

```bash
git push origin feature/my-feature

# 创建 PR
gh pr create --title "feat: 实现某功能" --body "描述..."
```

## PR 审查流程

### 自动检查项

PR 创建后，GitHub Actions 自动执行：

#### 阶段 1：代码质量
| 检查项 | 工具 | 说明 |
|--------|------|------|
| 格式检查 | Black | Python 代码格式 |
| Lint 检查 | Flake8 | 代码风格检查 |
| 类型检查 | MyPy | 静态类型检查 |

#### 阶段 2：测试
| 检查项 | 范围 | 说明 |
|--------|------|------|
| 单元测试 | 全量 | 所有 backend/tests/ |
| 集成测试 | 全量 | API 集成测试 |
| E2E 测试 | 轻量 | 只跑关键路径 |

#### 阶段 3：安全与部署
| 检查项 | 工具 | 说明 |
|--------|------|------|
| 安全扫描 | Trivy | 漏洞扫描 |
| Preview 部署 | Vercel | 前端预览环境 |

### 检查结果报告

PR 评论中会显示所有检查结果：

```
## ✅ PR 检查结果

- 代码质量: ✅
- 单元测试: ✅
- 安全扫描: ✅

---
*请确保所有检查通过后再请求审查*
```

### 审查要求

- ⚠️ 所有 CI 检查必须通过
- ⚠️ 至少 1 个 Reviewer 批准
- ⚠️ 解决所有对话

## Staging 环境（合并后自动）

### 触发条件

- PR 合并到 master/main 分支

### 自动执行流程

```
合并 → 后端部署 → 前端部署 → 完整E2E测试 → 性能测试 → 通知
```

### 完整 E2E 测试

| 测试类型 | 覆盖范围 |
|----------|----------|
| 创建任务 | 表单提交、验证、清空 |
| 任务列表 | 展示、筛选、排序 |
| 任务完成 | 状态更新、子任务检查 |
| 任务拆解 | 子任务创建 |
| 任务删除 | 级联删除 |

### 失败处理

如果 E2E 测试失败：
1. 自动创建 Issue（带 bug 标签）
2. 通知触发人
3. 提供 Artifacts 下载（测试报告、截图）

## 生产部署流程

### 部署到生产

1. 进入 GitHub Actions 页面
2. 选择 `deploy-production` workflow
3. 点击 "Run workflow"
4. 选择 `production` 环境
5. 勾选确认框
6. 执行部署
7. 等待健康检查通过

### 回滚操作

如果生产环境出现问题：

1. 进入 GitHub Actions 页面
2. 选择 `rollback` workflow
3. 点击 "Run workflow"
4. 选择回滚服务：
   - `backend` - 仅回滚后端
   - `frontend` - 仅回滚前端
   - `both` - 回滚全部
5. （可选）指定部署 ID
6. 确认执行

## 应急操作

### 紧急修复流程

```bash
# 1. 基于 master 创建 hotfix 分支
git checkout master
git pull origin master
git checkout -b hotfix/urgent-fix

# 2. 修复并本地测试
# ... 编码 ...
pytest tests/ -v
npx playwright test

# 3. 推送并创建紧急 PR
git push origin hotfix/urgent-fix
gh pr create --title "hotfix: 紧急修复" --body "..."

# 4. 快速审查后合并
# 5. 立即部署生产
```

### 回滚决策

| 现象 | 行动 |
|------|------|
| 健康检查失败 | 立即回滚 |
| 用户报告严重问题 | 评估后回滚 |
| 性能下降 > 50% | 考虑回滚 |
| 功能性 Bug | 修复后部署 |
